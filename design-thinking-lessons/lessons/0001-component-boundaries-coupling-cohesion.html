Where does this component end?

Why component boundaries are the decision every later architecture choice inherits.

Every downstream decision in this course — where state lives, whether a piece renders on the server, how a page scales — is constrained by a decision you make first, often without noticing you're making it: where you draw the edge of a component. Get the edge wrong and every later decision fights it. This lesson gives you the vocabulary to see the edge, and to defend where you drew it.

The symptom: a component that grew a permission system

Here's a component that looks completely ordinary — the kind you've written a hundred times under deadline:

// components/UserProfileCard.tsx
type UserProfileCardProps = {
  name: string;
  avatarUrl: string;
  bio: string;
  isAdmin: boolean;
  isCurrentUser: boolean;
  showEditButton: boolean;
  showDeleteButton: boolean;
  compact: boolean;
  onEdit?: () => void;
  onDelete?: () => void;
  theme: "light" | "dark";
};

export function UserProfileCard({
  name, avatarUrl, bio, isAdmin, isCurrentUser,
  showEditButton, showDeleteButton, compact, onEdit, onDelete, theme,
}: UserProfileCardProps) {
  return (
    <div className={`card ${theme} ${compact ? "card--compact" : ""}`}>
      <img src={avatarUrl} alt={name} />
      <h3>{name}</h3>
      {!compact && <p>{bio}</p>}
      {isAdmin && <span className="badge">Admin</span>}
      {(showEditButton || isCurrentUser) && <button onClick={onEdit}>Edit</button>}
      {showDeleteButton && isAdmin && <button onClick={onDelete}>Delete</button>}
    </div>
  );
}

Nothing here is a bug. It works. But look at what the prop list is actually doing: name, avatarUrl, bio, and theme answer "what does this look like?" — while isAdmin, isCurrentUser, showEditButton, and showDeleteButton answer an entirely different question: "who is allowed to do what?" One component is doing layout and permission logic. That's two unrelated jobs stitched into one interface.

Naming the problem: cohesion and coupling

Cohesion asks: how related are the things a single component is responsible for? A component with high cohesion does one job; every prop and every branch inside it serves that one job. UserProfileCard has low cohesion — rendering a card shell and deciding who may edit or delete a user are unrelated concerns, forced into the same box.

Coupling asks: how much does one piece of code have to know about another piece's internals to work with it? Every call site of UserProfileCard now has to know the exact boolean combination that produces the right button state — showEditButton || isCurrentUser is permission logic that leaked out of the component and into every caller, and it will drift out of sync the moment someone adds a caller who forgets the combination. That's tight coupling: change the permission rule, and you're now editing (or breaking) every place that constructs these props.

Cohesion and coupling aren't a React-specific idea — they come from structured design theory in the 1970s (Constantine & Yourdon). We're using them here on a component; later in this course we'll come back to the general version and use the same two words on backend and database code. Cohesion and coupling in React →

The fix: composition instead of configuration

The permission logic belongs where the permission knowledge already lives: the caller. So don't ask the card to accept flags describing every possible caller — ask it to accept the pieces a caller wants to arrange. This is the compound component pattern: one parent component plus a set of named sub-components that share an implicit contract, composed by whoever calls them.

// components/Card.tsx
export function Card({ compact, theme, children }: {
  compact?: boolean;
  theme: "light" | "dark";
  children: React.ReactNode;
}) {
  return (
    <div className={`card ${theme} ${compact ? "card--compact" : ""}`}>
      {children}
    </div>
  );
}

Card.Avatar = function CardAvatar({ src, alt }: { src: string; alt: string }) {
  return <img src={src} alt={alt} />;
};

Card.Body = function CardBody({ children }: { children: React.ReactNode }) {
  return <div className="card-body">{children}</div>;
};

Card.Actions = function CardActions({ children }: { children: React.ReactNode }) {
  return <div className="card-actions">{children}</div>;
};

And at the call site — the place that already knows the permission rule:

<Card theme="dark">
  <Card.Avatar src={user.avatarUrl} alt={user.name} />
  <Card.Body>
    <h3>{user.name}</h3>
    <p>{user.bio}</p>
  </Card.Body>
  {(isAdmin || isCurrentUser) && (
    <Card.Actions>
      <EditButton onClick={handleEdit} />
      {isAdmin && <DeleteButton onClick={handleDelete} />}
    </Card.Actions>
  )}
</Card>

Card is now highly cohesive — its only job is arranging a card shell — and loosely coupled to its callers, since it exposes slots (children, named sub-components) rather than a growing list of booleans that encode caller-specific logic. This is composition over configuration: instead of adding a flag for every future case, you give the caller pieces to assemble. How this constrains what's next

Card now holds no state and no interactivity of its own — that boundary decision is what will let Lesson 3 ask "does Card even need to run on the client?" (it won't; only EditButton/DeleteButton need an event handler). And the question of where isAdmin and isCurrentUser should live — local to the page, lifted, or derived from a session object — is exactly Lesson 2's subject. The boundary you draw today is the input to both of those decisions, not a separate concern.

Check your vocabulary

The original component bundled rendering layout, deciding who can edit, and deciding who can delete into one prop list. What term names a component doing multiple unrelated jobs?

After the refactor, a change to "who can delete" no longer requires editing Card. What term names components that communicate through minimal, well-defined props rather than encoding each other's internal logic?

The caller now assembles Card, Card.Avatar, Card.Body, and Card.Actions instead of toggling booleans inside one component. What pattern name applies to this API shape?

Secondary track: writing the decision down

You just made an architecture decision — composition over a growing boolean-flag API — for a reason a teammate reviewing your PR won't see just by reading the diff. This is what an ADR (Architecture Decision Record) is for: a short note capturing the context, the decision, and the consequences, so the "why" survives the code review and the next person who's tempted to add one more boolean flag can read why that path was already rejected. adr.github.io → is the canonical reference and template source — we'll come back to fuller ADRs later; for now you're just practicing the shape.

A short document recording the context, the decision made, and its consequences — used to justify a design choice like this one — is usually called a(n) ___.

Reveal a potential answer

The problem: type, isUnread, and compact answer "what does this look like?"; showDismiss and showMarkRead answer "what can the caller do here?" — two unrelated jobs in one prop list (low cohesion), and every caller has to know the right flag combination for its context (tight coupling). The refactor: a Notification shell that only handles layout, plus a slot (e.g. Notification.Actions) where the caller places whichever buttons apply — the same compound-component move as Card. The ADR: Context — the flag list grows with every new action; Decision — move action buttons into a caller-supplied slot; Consequences — callers assemble their own actions, but the component stops changing every time an action rule changes.
