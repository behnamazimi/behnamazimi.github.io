Where does this state live?

The boundary you drew in Lesson 1 already answered half of this question.

Carried over from Lesson 1

Lesson 1 made Card hold no state at all — it just arranges slots. That wasn't only a coupling fix. It also means every state question about a card ("is this the current user?", "is this admin?", "is the user following this profile?") now belongs to whoever calls Card, not to Card itself. Component boundaries and state placement are the same decision viewed from two angles.

The symptom: two booleans that disagree

Say a profile page shows a FollowButton in two places — the header, and a "suggested profiles" row in the sidebar, both for the same user:

// components/FollowButton.tsx
"use client";
export function FollowButton({ initialIsFollowing }: { initialIsFollowing: boolean }) {
  const [isFollowing, setIsFollowing] = useState(initialIsFollowing);
  return (
    <button onClick={() => setIsFollowing(!isFollowing)}>
      {isFollowing ? "Following" : "Follow"}
    </button>
  );
}

Click "Follow" in the header, and the sidebar instance still says "Follow." Both are true, both are wrong. Each FollowButton holds its own copy of a fact that should have exactly one answer.

Naming the problem: duplicated state

React's own docs name this directly: avoid duplication in state — when the same fact is tracked in more than one place, the copies will eventually disagree, because nothing forces them to update together. Choosing the State Structure — react.dev → The fix isn't a better useState call — it's deciding where the one true copy of "is this user followed?" should live, then having every consumer read from it instead of keeping its own.

Three places state can live — and how to choose

1. Local state — when nothing else needs to know

A dropdown's open/closed flag, an input's draft value before submit — nothing outside that component cares. Keep it local; this is the default, not a fallback.

2. Lifted / shared state — when siblings need one answer

Here, both FollowButton instances need the same answer. The fix is lifting state up: move isFollowing to the closest common ancestor of both buttons, and pass it down. If the two buttons are far apart in the tree (a header vs. a sidebar in different layout branches), "closest common ancestor" might be a page-level context or a small client-side store (Zustand, Context) — same principle, just a longer reach. Nadia Makarevich's 2025 framework for this: default to lifting; only reach for a library when the "closest common ancestor" is unreasonably far up the tree.

// app/profile/[id]/page.tsx — the common ancestor now owns the fact
export default function ProfilePage({ user, initialIsFollowing }: Props) {
  const [isFollowing, setIsFollowing] = useState(initialIsFollowing);
  return (
    <>
      <ProfileHeader user={user}>
        <FollowButton isFollowing={isFollowing} onToggle={setIsFollowing} />
      </ProfileHeader>
      <SuggestedRow user={user}>
        <FollowButton isFollowing={isFollowing} onToggle={setIsFollowing} />
      </SuggestedRow>
    </>
  );
}

3. Derived state — when you can compute it instead of storing it

Now suppose the header also shows "1,204 followers." Don't store followerCountLabel in its own useState — it's a formatting of followerCount, computed fresh every render:

const followerCountLabel = new Intl.NumberFormat().format(followerCount) + " followers";

The moment you catch yourself writing a useEffect whose only job is to keep one piece of state in sync with another, that's the tell: one of them shouldn't be state at all — it should be derived at render time from the other.

Note isFollowing itself is arguably not really client state either — it's a fact about a relationship stored server-side. We're treating it as lifted client state here for the lifting-vs-duplication lesson, but Lesson 3 revisits where that fact should actually be read from, once request-time server data is on the table.

Secondary track: what this does to your tests

The moment isFollowing moves from local to lifted/shared, testing FollowButton in isolation stops proving much — you'd need to fake the parent's state and the props contract just to see it render. This is exactly why Kent C. Dodds's Testing Trophy weights integration tests over isolated unit tests for UI: "the more your tests resemble the way your software is used, the more confidence they give you," and shared state is used across components, not inside one. A test that renders ProfilePage and clicks both buttons proves the sync works; a unit test of FollowButton alone can't.
