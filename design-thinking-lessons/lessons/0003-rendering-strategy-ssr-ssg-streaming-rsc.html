When does this actually run?

Component boundaries decided what ships to the client. Rendering strategy decides when each piece's work happens.

Carried over from Lessons 1 & 2

Card has no state or interactivity — Lesson 1's boundary decision. isFollowing is lifted client state that's actually a fact about a server-side relationship — Lesson 2's open question. Both land here: the boundary decides which pieces can stay Server Components; the state's real origin (server data) decides which piece must run at request time.

First, a vocabulary correction

A lot of 2026-era tutorials still describe SSR, SSG, and ISR as three separate modes you pick per page. That's stale. In the current Next.js App Router model, the choice isn't per-page — it's per-component, driven by whether a component opts into caching, and the result is stitched together at request time. This isn't a small nuance: using "I set this page to SSG" as if it were still a single per-page toggle is exactly the kind of correct-idea-wrong-vocabulary mismatch worth catching in yourself. Verify against the current caching docs → before trusting any blog post's terminology here, including this one in a year.

The default: Server Components

Every component in the App Router is a Server Component unless it opts out. Server Components run only on the server — they can query a database directly, never ship their own code to the browser, and their output is just HTML (plus a serialized description React uses for updates). A component becomes a Client Component only when it adds "use client" — which it needs the moment it uses state, effects, or event handlers. Server Components — react.dev → This is exactly the line Lesson 1 drew: Card needed none of those, so it stays a Server Component; FollowButton needs an onClick, so it must be a Client Component.

The case study: one profile page, two very different needs

ProfileHeader (name, avatar, bio) rarely changes and isn't specific to who's viewing it — a great candidate to cache. Whether the viewer is following this profile is specific to the viewer and must be current — it can't be baked in ahead of time. Here's both, current syntax, verified against the Next.js docs:

// app/profile/[id]/page.tsx
import { Suspense } from "react";
import { cookies } from "next/headers";
import { cacheLife } from "next/cache";

export default async function ProfilePage({
  params,
}: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  return (
    <>
      <ProfileHeader userId={id} />
      <Suspense fallback={<FollowButtonSkeleton />}>
        <FollowButtonForViewer profileId={id} />
      </Suspense>
    </>
  );
}

// Cached: built once, reused across viewers, refreshed hourly.
async function ProfileHeader({ userId }: { userId: string }) {
  "use cache";
  cacheLife("hours");
  const user = await getUser(userId);
  return (
    <Card theme="light">
      <Card.Avatar src={user.avatarUrl} alt={user.name} />
      <Card.Body><h3>{user.name}</h3><p>{user.bio}</p></Card.Body>
    </Card>
  );
}

// Never cached: reads the viewer's own cookie, must be current for THIS request.
async function FollowButtonForViewer({ profileId }: { profileId: string }) {
  const viewerId = (await cookies()).get("uid")?.value;
  const isFollowing = await getIsFollowing(viewerId, profileId);
  return <FollowButton profileId={profileId} initialIsFollowing={isFollowing} />;
}

ProfileHeader is doing the job the old mental model called "SSG" (or "ISR" once it refreshes) — computed ahead of need, reused. FollowButtonForViewer is doing the job the old mental model called "SSR" — computed fresh, per request, because it depends on who's asking. The difference now is that both live on the same page, as a property of each component, not a setting on the route.

Streaming and Partial Prerendering

Because FollowButtonForViewer is wrapped in <Suspense>, the browser doesn't wait for it: the cached ProfileHeader plus the Suspense fallback go out immediately as a static shell, and the real follow-button content streams in once it resolves. A page assembled this way — a cached static shell with dynamic Suspense-wrapped holes — is doing Partial Prerendering (PPR), the current default rendering shape in Next.js. How to Choose the Best Rendering Strategy — Vercel Blog → is the primary source for this lesson; it's written by the framework's own team and lays out the tradeoff table this lesson only has room to summarize.

Secondary track: when caching becomes an incident

Notice what ProfileHeader's cache key implicitly depends on: userId. If a refactor accidentally cached a component keyed on the wrong argument — say, caching FollowButtonForViewer's output without viewerId in the key — one visitor's "is following" state could be served to a different visitor. This is a real, common class of production incident: a caching boundary mismatch that leaks or misattributes user-specific data. Two vocabulary items worth having ready before it happens to you: blast radius (how many users/requests were affected) and blameless postmortem (the writeup that focuses on the contributing cause, not who wrote the bad cache key). // app/products/[id]/page.tsx
export default async function ProductPage({ params }: Props) {
  "use cache";
  const { id } = await params;
  const product = await getProduct(id);
  const viewerId = (await cookies()).get("uid")?.value; // ⚠️
  const price = await getPriceForViewer(product.id, viewerId); // member discount
  return (
    <>
      <ProductInfo product={product} />
      <PricePanel price={price} />
    </>
  );
}

Reveal a potential answer

What's wrong: because the whole page is "use cache"'d, the first viewer's member-discount price gets baked into the cached output and served to every later viewer of that product page — a price (and effectively a discount eligibility) leak between users. The fix: keep ProductInfo cached (it's the same for everyone); move PricePanel into its own component that reads the viewer's cookie directly and sits inside a Suspense boundary, so it's computed fresh per request instead of baked into the cache. The ADR: Context — price depends on a per-viewer cookie but the whole page is cached; Decision — split into a cached shell plus a request-time Suspense-wrapped price component; Consequences — the price is always correct per viewer, at the cost of one extra request-time render for that piece.
