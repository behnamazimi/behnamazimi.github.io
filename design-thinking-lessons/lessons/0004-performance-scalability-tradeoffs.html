What does this cost, and who pays?

Three lessons of boundary decisions land here as a bill — measured in bytes, milliseconds, and server load.

Carried over from Lessons 1–3

Every "use client" boundary from Lesson 1 (only FollowButton, not Card) and every cache decision from Lesson 3 (ProfileHeader cached, follow-state not) already set most of this lesson's outcome before you read a single performance metric. This lesson gives you the vocabulary to see the bill, not a new set of decisions to make from scratch.

Two different questions, often confused

Performance: how fast is this for one user, right now? Scalability: what happens to cost and latency as load grows — more users, more data, more traffic? A page can be fast for you in dev and still collapse at 10,000 concurrent users if every request does uncached work. Lesson 3's caching decision is exactly what buys scalability: a cached ProfileHeader costs roughly the same whether one person or one million people request it; an uncached per-request computation scales roughly linearly with traffic.

The bill for a Client Component: bundle size and hydration

Every "use client" component adds its code — and the code of everything it imports — to the JavaScript the browser must download, parse, and execute before the page is interactive. That total is bundle size. Once downloaded, React has to walk the server-rendered HTML and attach event handlers and state to it — that process is hydration, and its cost scales with how much of the tree is client-side, not with how much is visible on screen. This is precisely why Lesson 1's boundary mattered beyond coupling. Compare:

// Costly: the whole card ships to the client for one button's sake
"use client";
export function Card({ children }: { children: React.ReactNode }) {
  return <div className="card">{children}</div>;
}
// Avatar, Body, and every future addition now hydrate — whether or not
// they ever need to.

// Actual (Lesson 1's boundary): only the interactive leaf opts in
export function Card({ children }: { children: React.ReactNode }) {
  return <div className="card">{children}</div>; // Server Component
}
// FollowButton.tsx
"use client";
export function FollowButton(props: Props) { /* ... */ }

Marking Card as a client component doesn't make it work any differently today — it works either way. The cost shows up later, as every future avatar, badge, or layout tweak added to Card silently grows the client bundle, because the boundary was drawn one level too high.

Naming what you're actually measuring: Core Web Vitals

"Fast" isn't one number. Google's Core Web Vitals name three specific axes: LCP (Largest Contentful Paint — how long until the main content is visible), INP (Interaction to Next Paint — how long until a click or keypress gets a visible response), and CLS (Cumulative Layout Shift — how much content jumps around as it loads). Web Vitals — web.dev →, the primary source for this lesson. A bloated client bundle mainly hurts INP (more JS to parse and run before the thread is free to respond); an unbounded uncached database query mainly hurts scalability under load, which shows up as LCP degrading only once traffic rises — a failure mode a single-user dev-server test will never reveal.

Secondary track: this is a code review smell

Google's code review standard asks reviewers to weigh whether a change's added complexity is justified — and a new "use client" boundary is exactly that kind of complexity: it's easy to add, and its cost (bundle size, hydration) doesn't show up in the diff itself. The Standard of Code Review — Google eng-practices → A reviewer who's fluent in this lesson's vocabulary asks "does this whole subtree need to hydrate, or just the one interactive leaf?" — a question that's hard to ask without the words boundary, bundle size, and hydration on hand.

export function ArticleList({ articles }: { articles: Article[] }) {

Reveal a potential answer

What's wrong: "use client" on ArticleList pulls the entire list — every title, every summary — into the client bundle and hydration path, just so one deeply nested ReadMoreToggle can hold a boolean. The fix: move "use client" down to ReadMoreToggle itself; ArticleList and ArticleSummary stay Server Components and ship no JS of their own. The vital: mainly INP — less JavaScript has to parse and run before the page responds to input — with a secondary win on bundle size generally.
