Lesson 0005 · Track: Frontend Architecture (added topic) · ~15 min

Why I added this one

Component boundaries, state, rendering, and performance all meet at one practical question: where does the data actually come from, and when?

Why this earned a slot

Lesson 3 gave every component the right to fetch its own data — Server Components can fetch anywhere in the tree, not just at the page's top. That freedom has a cost: if you don't sequence fetches deliberately, the same boundary that made your code cleaner can silently make your page slower. This is the connective-tissue topic between "the boundaries are right" (Lessons 1–3) and "it's still slow" (Lesson 4).

The symptom: a waterfall

Nested components each fetching their own data looks clean — until the fetches depend on nothing but still run one after another:

// Naive: sequential even though nothing requires it
export default async function Page({ params }: Props) {
  const { username } = await params;
  const artist = await getArtist(username);   // waits...
  const albums = await getAlbums(username);    // ...then waits again
  return <div>{artist.name}</div>;
}

getAlbums doesn't need anything from artist — it only needs username, which was already available. But because it's written after an await, it doesn't start until getArtist finishes. Two independent round-trips just became one twice as long. This exact case is documented directly by Next.js: Fetching Data — Next.js docs →, the primary source for this lesson, verified against the current v16 docs rather than an older tutorial.

The fix: parallel when independent, Suspense when genuinely dependent

async function getArtist(username: string) {
  const res = await fetch(`https://api.example.com/artist/${username}`);
  return res.json();
}
async function getAlbums(username: string) {
  const res = await fetch(`https://api.example.com/artist/${username}/albums`);
  return res.json();
}

export default async function Page({ params }: Props) {
  const { username } = await params;
  // Both calls start immediately — neither needs the other's result.
  const artistData = getArtist(username);
  const albumsData = getAlbums(username);
  const [artist, albums] = await Promise.all([artistData, albumsData]);
  return (
    <>
      <h1>{artist.name}</h1>
      <Albums list={albums} />
    </>
  );
}

Compare this to a request that's actually dependent — fetching playlists that need the artist's id, which only exists after getArtist resolves:

export default async function Page({ params }: Props) {
  const { username } = await params;
  const artist = await getArtist(username); // genuinely can't be parallel
  return (
    <>
      <h1>{artist.name}</h1>
      <Suspense fallback={<div>Loading...</div>}>
        <Playlists artistID={artist.id} />
      </Suspense>
    </>
  );
}

The sequencing here is unavoidable, but <Suspense> — Lesson 3's tool — keeps the artist's name visible immediately instead of blocking the whole page on the playlists request. Sequential-but-necessary and sequential-by-accident look identical in a stack trace; the difference is entirely in whether a later request's arguments depend on an earlier one's result.

The other half: not fetching the same thing twice

Lesson 2 named "duplicated state" as a bug. The fetching equivalent isn't a bug in the same way — Next.js's extended fetch automatically memoizes identical requests within one render pass, and React.cache extends the same dedup to non-fetch sources like a database client:

import { cache } from "react";
export const getUser = cache(async (id: string) => {
  const res = await fetch(`https://api.example.com/user/${id}`);
  return res.json();
});
// Called from two different components in the same request → one real call.

This is why Lesson 3's "components can fetch their own data" model doesn't cause duplicate network calls the way duplicated useState caused duplicated facts — the memoization layer handles it, as long as the request shape (function + arguments) is actually identical.

Secondary track: waterfalls hide from unit tests

A unit test that mocks getArtist and getAlbums as already-resolved promises can't tell you whether they ran sequentially or in parallel in production — the mock resolves instantly either way. This is another point for Lesson 2's Testing Trophy bias toward integration tests: only something that exercises real (or realistically delayed) async timing, or a request-count assertion, actually catches a waterfall regression.

Reveal a potential answer

Dependent vs. independent: getProduct(id) has to run first — nothing else has an id to work with yet. getReviews(product.id) and getRelatedProducts(product.category) both need a field off product, but not off each other — so once product resolves, they can run in parallel. The rewrite: await getProduct(id), then call both getReviews and getRelatedProducts before awaiting either, and resolve them together with Promise.all. The term: the original version was a data waterfall — two independent calls forced into sequence for no reason.
