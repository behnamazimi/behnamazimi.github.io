If coupling and cohesion only mean something inside a .tsx file, you haven't actually learned the principle — you've memorized where it shows up once.

Where this comes from

Lesson 1 used cohesion and coupling to fix a React component. This lesson strips the React away and asks: what's the actual, language-independent idea — and does it show up the same way in your PHP and MySQL work?

The general version: deep modules, not just cohesive components

John Ousterhout's framing in A Philosophy of Software Design generalizes exactly what Lesson 1 did with Card: a module's value is its functionality divided by the complexity of its interface. A deep module has a simple interface hiding substantial complexity. A shallow module has an interface about as complex as what it actually does — you're paying interface-complexity tax without buying much hidden-complexity benefit. The mechanism that makes a module deep is information hiding: callers depend on what a module promises, not on how it's built inside. A Philosophy of Software Design — John Ousterhout →, the primary source for this lesson.

Card from Lesson 1, after the refactor, is a deep module: three props and a slot API hide all of the layout mechanics. The original UserProfileCard was shallow: eleven props whose combinations you had to understand just to use it safely — the interface was nearly as complicated as the behavior it produced.

The same shape in PHP

Here's a repository class with the identical disease, no JSX anywhere:

<?php
final class OrderRepository
{
    private ?string $statusFilter = null;
    private ?DateTimeImmutable $dateFrom = null;
    private ?DateTimeImmutable $dateTo = null;

    public function setStatusFilter(string $status): void { $this->statusFilter = $status; }
    public function setDateFrom(DateTimeImmutable $d): void { $this->dateFrom = $d; }
    public function setDateTo(DateTimeImmutable $d): void { $this->dateTo = $d; }

    public function execute(): array
    {
        // Builds SQL from whichever setters happened to be called, in whatever order.
    }
}

// Caller must know the right setter sequence before execute() behaves correctly —
// the exact same problem as UserProfileCard's boolean combinations.
$repo = new OrderRepository();
$repo->setStatusFilter('paid');
$repo->setDateFrom($from);
$result = $repo->execute();

This is shallow for the same reason UserProfileCard was: the interface (three setters plus an order-dependent execute) doesn't hide complexity, it just relocates it into "which setters did you remember to call." The deep alternative collapses the whole thing into one call whose argument is the interface:

<?php
final class OrderRepository
{
    public function findByFilters(OrderFilters $filters): array
    {
        // Query construction, indexing strategy, pagination — all hidden here.
    }
}

$result = $repo->findByFilters(new OrderFilters(status: 'paid', from: $from));

One method. The caller states what it wants; it never needs to know how the SQL gets built, in what order, or with what internal state. That's information hiding doing the same job Card.Avatar / Card.Body / Card.Actions did for a UI tree.

Secondary track: this is a review heuristic in any language

"Is this deep or shallow?" is a code-review question independent of framework — it works the same on a PHP repository, a MySQL stored procedure, or a TypeScript hook. Once you have the vocabulary, you can ask a reviewer (or yourself) the Lesson 4 question — "is this complexity justified?" — with a precise answer instead of a vague feeling that something's "messy."

How to tell if it landed

There's no single correct example here — it's your own code. The test: does describing your "shallow" example require a caller to remember a specific sequence or combination of calls before it behaves correctly (like OrderRepository's setters)? And does your "deeper" version collapse that into one call whose argument states the intent directly (like findByFilters)? If both hold, the generalization landed. If you could only describe the shallow example using React terms, it hasn't yet — that's the real test of this broadening pass.
