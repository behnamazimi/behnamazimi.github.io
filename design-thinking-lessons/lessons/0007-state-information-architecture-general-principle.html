Every "where should this live" question is the same question

React didn't invent single-source-of-truth. Your MySQL schema has been asking you this since before you touched React.

Where this comes from

Lesson 2 fixed two FollowButtons disagreeing by giving isFollowing exactly one home. That's not a React idea wearing a React costume — it's the same problem a database schema solves with normalization, and the same problem a cached column solves (or mishandles) with denormalization.

The same bug, in a schema instead of a component tree

-- customers
CREATE TABLE customers (id INT PRIMARY KEY, email VARCHAR(255));

-- orders — email duplicated instead of referenced
CREATE TABLE orders (
  id INT PRIMARY KEY,
  customer_id INT,
  customer_email VARCHAR(255), -- ⚠️ same fact, second home
  total DECIMAL(10,2)
);

Update a customer's email, and every existing row in orders still has the old one. This is exactly Lesson 2's duplicated-state bug — the same fact tracked independently in two places, nothing forcing them to agree — just wearing SQL instead of JSX. The database-native name for organizing schema so each fact has one home is normalization: orders should hold customer_id and join to customers for the email, the same way Lesson 2's page held one isFollowing value and passed it down instead of letting each button keep its own. Database Normalization — IBM →, primary source for this lesson.

Derived state has a database twin too — with a real tradeoff attached

Lesson 2's followerCountLabel was computed at render time, never stored. The database equivalent is a plain VIEW — computed fresh on every query, never stale, but potentially slow if the underlying join is expensive. Sometimes, deliberately, you trade that freshness for speed: a cached order_count column on customers, refreshed on a schedule instead of computed live. That's denormalization — and unlike the accidental customer_email duplication above, it's a real engineering tradeoff, not a bug, as long as you know what you just signed up for: the cached value can now be stale between refreshes. Why, How, and When to Use Materialized Views — Materialize → lays out this exact tradeoff (note: MySQL has no native materialized view — the usual workaround is a real table populated by INSERT...SELECT and refreshed on a schedule or trigger).

customer_email is stored independently in both customers and orders, with nothing keeping the copies in sync. What database principle does this violate?

A cached order_count column on customers, refreshed on a schedule instead of computed fresh from orders every time. What's this called?

The delay between when the real data changes and when a cached or denormalized copy catches up. What's this called?

Secondary track: denormalizing on purpose deserves an ADR

Adding a cached order_count column is a real architecture decision with a real cost (a trigger or cron job to maintain, a staleness window someone has to accept). It's exactly the shape Lesson 1 introduced: Context (live count query too slow at this table size), Decision (cache the count, refresh via trigger), Consequences (up to N minutes of staleness; one more thing that can silently break). adr.github.io →

Your turn: is this actually duplication?

CREATE TABLE order_items (
  id INT PRIMARY KEY,
  order_id INT,
  product_id INT,
  product_name VARCHAR(255),   -- same fact as products.name?
  unit_price DECIMAL(10,2)     -- same fact as products.price?
);

Exercise

products already has name and price columns. Before you call this duplication, think carefully. In the scratchpad:

  1. Is order_items.product_name / unit_price the same fact as products.name / price, or a different fact that happens to start out equal? (Hint: what happens to each one over the following year?)
  2. Based on that, is storing them here a normalization bug, or the correct call? Justify it.

Check your answer

This exercise has a real answer, not an "it depends" shrug — make sure you've committed to one before expanding this.

Reveal the answer

They're not duplication — they're different facts that happen to start out equal. products.name/price are the current catalog values; order_items.product_name/unit_price are a snapshot of what was true at the moment of purchase. Over the following year the catalog price can change, but the receipt must not — that's the entire point of storing it separately. Storing both here is the correct call, not a normalization bug.
