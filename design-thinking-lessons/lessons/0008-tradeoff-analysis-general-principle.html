"It depends" is not an answer — naming the axes is

Every rendering choice in Lesson 3 was a freshness-vs-cost tradeoff wearing a Next.js costume.

Where this comes from

Lesson 3's ProfileHeader chose to be cached (cheap, fast, briefly stale). Its FollowButtonForViewer chose to be computed live (always current, costs a request every time). Neither choice is universally right — each is a deliberate trade of freshness against cost. That trade has a name outside frontend, and it's one of the most famous in distributed systems.

The general shape: CAP theorem

Eric Brewer's CAP theorem, stated for distributed data stores facing a network partition: you can't simultaneously guarantee Consistency (every read gets the latest write, or an error), Availability (every request gets a non-error response, even if not the latest data), and Partition tolerance (the system keeps working despite dropped messages between nodes). When a partition happens, you pick — consistency or availability; you don't get both. CAP Theorem Explained — BMC →, primary source for this lesson.

The literal theorem is about distributed databases. The transferable skill is the shape of the reasoning: name the specific things in tension, then state — out loud, in writing — which one you're deliberately giving up and why. That's tradeoff analysis. "It depends" is what you say before you've done it; naming the axes is what doing it looks like.

Mapping Lesson 3 onto this shape

ProfileHeader's "use cache" chose availability-and-speed over strict consistency: it will happily serve a slightly outdated bio rather than block or error. FollowButtonForViewer chose consistency over raw speed: it pays a request every time specifically so the answer is never wrong for the viewer asking. Same page, two different axis choices, both correct — because they're answering different questions about different data.

Check your vocabulary

Every read receives the most recent write, or an error — never silently stale data. What's this guarantee called?

Every request receives a non-error response, without any guarantee it reflects the very latest write. What's this guarantee called?

Secondary track: this is the "Alternatives Considered" section

By now you've sketched a couple of ADRs across this course. The part of an ADR that's easiest to skip — and the part that actually proves you did tradeoff analysis instead of just picking the first option — is naming the alternative you rejected and why. adr.github.io → If your Consequences section only lists upsides, you haven't written down the tradeoff — you've written down a decision with no analysis attached.

Exercise

Two features on the same e-commerce product page:

  • A "12 people viewed this today" counter.
  • The "3 left in stock" inventory count shown right before checkout.

In the scratchpad:

  1. For each, name which side of the consistency/availability tension you'd deliberately pick, and why the other one would actually hurt the business here.
  2. State it as a one-line tradeoff statement for each ("We chose ___ over ___ because ___") — not "it depends."

Check your answer

Check both one-liners against this — if either one is "it depends" dressed up, that's the thing to fix before moving on.

Reveal a potential answer

View counter: we chose availability over consistency because a slightly stale count is harmless, and the counter must never block or error just to stay perfectly accurate. Inventory count at checkout: we chose consistency over availability because serving a wrong "in stock" answer here means overselling — better to show an error or ask the shopper to recheck than let two people both "successfully" buy the last unit.
