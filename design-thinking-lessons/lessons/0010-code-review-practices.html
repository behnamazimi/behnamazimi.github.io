Lesson 0010 · Track: Process & Collaboration (Code review) · ~15 min

What is a review actually for?

Not "find every flaw." Not "make it perfect." Something narrower, and more useful.

Where this connects

Lessons 4 and 9 already reached for this track once each — "is this "use client" justified?", "is this complexity essential or accidental?" This lesson gives that instinct its own foundation, instead of leaving it as a footnote.

The purpose, stated precisely

Google's engineering practices state it directly: the primary purpose of code review is making sure the overall code health of the codebase improves over time — not achieving a perfect change. A reviewer should favor approving a change once it definitely improves code health, even if it isn't perfect. The Standard of Code Review — Google eng-practices →, primary source for this lesson. This reframes what you're doing in a review: not gatekeeping for perfection, but checking a specific, narrower thing — does this leave the codebase better than it found it.

What to actually look for

The same standard names four things worth checking, in order of how much they matter: functionality (does it do what it's supposed to, and is that good for users), complexity (could this be simpler — will another engineer understand it later), tests (are they correct and well-designed), and naming (are things called what they are). Notice that "complexity" is exactly Lesson 9's question wearing a review hat: is this essential to the problem, or accidental complexity someone (maybe the author, maybe you reviewing your own draft) introduced?

How to say it

Comment on the code, never the developer. Reserve unqualified statements for things that actually block merging; prefix optional polish with Nit: so the author knows it's take-it-or-leave-it. When you disagree and can't resolve it in the thread, escalate — a quick call, then a tech lead or maintainer, rather than a comment war. Review is also where knowledge transfer happens: a comment that teaches something is worth writing even when it isn't blocking anything.

Reveal model review comments

Prop list: "This is heading toward the same low-cohesion/shallow-interface problem UserProfileCard had — display flags and caller-action flags are mixed together, and it'll keep growing with every new post action." "use client": "The whole card doesn't need to hydrate for one button's state — this is accidental complexity/bundle-size cost. Can we move "use client" down to just the pin/report buttons?" localPinned: "If this card ever renders twice for the same post (e.g. feed + a pinned-posts sidebar), these two copies of pinned state can disagree — duplicated state. Should this be lifted, or read from the server value instead?" Nit vs. blocking: the "use client" and localPinned comments are blocking — they're real bugs waiting to happen. The prop-list comment could be a Nit: only under real time pressure; otherwise it's worth blocking on too, since it predicts the next few PRs' worth of pain.
