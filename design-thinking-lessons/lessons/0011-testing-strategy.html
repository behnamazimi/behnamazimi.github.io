Which test actually earns its keep?

Every architecture decision so far quietly decided how testable its own code would be.

Where this connects

Lessons 2 and 5 each reached for this track once — shared state hurting isolated tests, waterfalls hiding from mocked ones. This lesson gives the whole shape a name.

The trophy, not the pyramid

Kent C. Dodds's Testing Trophy ranks test types by confidence-per-cost, not by count: static analysis (types, lint — catches typos and shape errors before anything runs), unit tests (one function or component, isolated), integration tests (several units working together, the way they actually get used), and E2E tests (a real browser against something close to production). His rule: "the more your tests resemble the way your software is used, the more confidence they can give you" — which is why the trophy is fattest in the middle, at integration, rather than at the base, unlike the older Testing Pyramid. The Testing Trophy — Kent C. Dodds →, primary source, and the same one Lessons 2 and 5 already pointed to.

Your architecture already picked your tests for you

Three places this course's own examples already showed it:

  • Lesson 1's Card is trivially unit-testable — render it with different children, assert the output — precisely because it's a deep, cohesive module with no hidden dependency on anything else. The original UserProfileCard, with eleven interdependent booleans, would need a combinatorial pile of unit tests just to cover its own interface.
  • Lesson 2's lifted isFollowing means a unit test of FollowButton alone can pass while the actual bug (two instances disagreeing) is invisible — you need an integration test that renders the common ancestor and both buttons together.
  • Lesson 3's Server Components are async functions that run only on the server; you can't hand one to React Testing Library's render() the way you would a Client Component. In practice this pushes verification toward integration tests (render the route, assert on output) or E2E — a real, current gap worth naming rather than discovering by surprise.

Exercise

Using the Lesson 1–3 ProfilePage (Card, FollowButton, FollowButtonForViewer), fill in one concrete test per layer in the scratchpad:

  1. Unit: one thing worth unit-testing in isolation, and why it's safe to (what makes it cohesive enough not to need its neighbors).
  2. Integration: one thing that requires rendering more than one component together — specifically, what bug would only show up at this layer.
  3. E2E: one thing that requires a real browser and can't be faked with mocks.

Saved locally in your browser as you type.

Check your answer

Check specifically whether your integration-test pick is catching something a unit test structurally can't — not just "a bigger unit test."

Reveal a potential answer

Unit: Card rendered in isolation with different children — safe because it's cohesive and has no dependency on anything outside itself. Integration: render the page and confirm FollowButtonForViewer shows the correct follow state for the current viewer's cookie — this is the layer that would actually catch a caching/cookie-key mistake like Lesson 3's; a unit test mocking the viewer ID away can't. E2E: click Follow in a real browser, reload the page, and confirm it's still following — this crosses the real server action, cookie, and database round-trip that no mock can fake.
