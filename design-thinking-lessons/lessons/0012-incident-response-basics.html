Where this connects

Lesson 3 named the risk: if ProfileHeader's cache key ever omitted a needed argument, one visitor's cached output could leak to another. Assume it happened. This lesson is the response, not the fix.

Severity first, fix second

Before touching code, name how serious this is — a severity level (often Sev1 highest to Sev4 lowest) that sets urgency and who needs to know. A UI glitch is low severity. A cache bug that shows one user's follow-relationships or private data to another is high severity regardless of how few lines of code caused it — severity tracks impact, not the size of the diff. Alongside severity, name the blast radius: how many users, requests, or how much data were actually affected. Lesson 3 already used this term; here's the retrieval check — if you had to guess without re-reading, would "blast radius" have come to mind unprompted?

Triage → mitigate → resolve

The order matters: first triage (confirm it's real, assess severity and blast radius), then mitigate (stop the bleeding — here, likely purging the bad cache entries or disabling caching for that component, not necessarily shipping the real fix yet), then resolve (the actual code fix, reviewed and deployed normally). Mitigation is deliberately allowed to be a blunt instrument — turning off caching entirely is an acceptable mitigation even though it costs the scalability Lesson 4 cared about, because right now correctness outranks it.

The write-up: blameless, not vague

Once resolved, the postmortem records what happened without assigning individual fault — blameless means it assumes everyone involved acted reasonably with the information they had, and focuses on contributing causes a process or system could catch next time. A useful postmortem still names the specific mechanism precisely (the cache key omitted viewerId) — blameless is about people, not about vagueness.

Reveal a model postmortem

What happened: ProfileHeader's cache key omitted viewerId, so a cached fragment computed for one viewer was served to others. Impact / blast radius: every viewer of a cached profile page during the roughly six-hour window before detection — the exact count depends on traffic, but the failure mode (one user's data reaching another) is serious enough to treat as high severity regardless of how few requests were involved. Root cause: the "use cache" function read viewerId from a cookie but didn't include it as an argument, so it wasn't part of the cache key. Action items: add a test asserting that any cached function's argument list includes every runtime value it reads (cookies, headers); add this as a standing code-review checklist item for new "use cache" usage.
