The document that would have caught Lesson 12's incident

Not every decision needs this. The ones with real tradeoffs and real risk do.

Where this connects

You've sketched four-line ADRs in Lessons 1, 3, 7, and 8 — Context / Decision / Consequences, one decision, quick. Lesson 3's caching architecture and Lesson 12's incident are exactly the case where that's not enough. This lesson is about when to reach for something bigger, and what "bigger" actually contains.

ADR vs. design doc — different jobs

An ADR records one decision, after the fact or right before it, tersely — Context, Decision, Consequences. A full design doc is written before building something with real tradeoffs, for an audience who wasn't in the room: it states the problem, what's explicitly in and out of scope, the proposed approach, the alternatives that were rejected and why, and the risks. adr.github.io → remains the primary source for the ADR half; the design-doc half doesn't have one canonical source the way ADRs do — treat the structure below as the practical minimum, not gospel from one author.

The section everyone skips, and the one that would have mattered here

A design doc for Lesson 3's ProfileHeader caching decision needs a Non-goals section — stating what this design explicitly isn't trying to solve, so scope doesn't silently creep — and an Alternatives considered section: cache with no TTL, cache with a short TTL, don't cache at all, cache but explicitly key by every runtime input. Writing that section down is what actually proves Lesson 8's tradeoff analysis happened, instead of just picking the first option that worked in dev. And a Risks section stating "if the cache key ever omits an argument, one user's data could leak to another" is precisely the sentence that — if written down before shipping — turns Lesson 12's incident from a surprise into a known, watched-for failure mode.

Reveal a model design doc

Problem: without caching, ProfileHeader re-queries the database on every request. Goals / Non-goals: goal — cache the parts of a profile that are identical for every viewer; non-goal — this doesn't address per-viewer personalization (follow status), which is handled separately. Proposed design: cache ProfileHeader with "use cache" keyed only by userId; stream the viewer-specific follow button separately via Suspense, reading the viewer's cookie at request time. Alternatives considered: caching the whole page including follow state — rejected, since that's precisely the mechanism behind Lesson 12's incident. Risks: if a future change makes ProfileHeader read anything else from cookies/headers, the cache key must be updated to include it, or one viewer's cached output could leak to another — add a test asserting the cache key covers every runtime value the function reads.
