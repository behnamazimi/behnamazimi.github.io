Component Boundaries Cheat Sheet

Terms

Cohesion
How related a component's own responsibilities are to each other. High cohesion = one job. Low cohesion = unrelated jobs sharing one interface.

Avoid as synonyms: "clean", "well-organized" — too vague to reason with.

Coupling
How much one piece of code must know about another's internals to use it correctly. Tight coupling = callers must replicate internal logic (e.g. a boolean combination). Loose coupling = callers depend only on a small, stable interface.

Avoid: "dependent" alone — say tightly or loosely coupled, and to what.

Composition over configuration
Give the caller pieces to assemble (children, named sub-components) instead of a growing list of flags that encode every caller's use case.
Compound component
A parent component plus named sub-components (Card.Avatar, Card.Body, Card.Actions) that share an implicit contract and are assembled by the caller.

Diagnostic: is this a boundary problem?

SignalUsually means
Prop count keeps growing across PRsLow cohesion — component is absorbing new jobs
Boolean props control unrelated behavior (showX, isY)Configuration is standing in for composition
Prop names describe who (isAdmin) rather than what to renderCaller-specific logic leaked into the component
Every caller needs the same boolean combination to get correct behaviorTight coupling — logic duplicated across call sites

Before / after (condensed)

// Before: configuration via flags — low cohesion, tight coupling
<UserProfileCard isAdmin isCurrentUser showEditButton showDeleteButton />

// After: composition via slots — high cohesion, loose coupling
<Card theme="dark">
  <Card.Avatar src={user.avatarUrl} alt={user.name} />
  <Card.Body>...</Card.Body>
  {(isAdmin || isCurrentUser) && <Card.Actions>...</Card.Actions>}
</Card>
