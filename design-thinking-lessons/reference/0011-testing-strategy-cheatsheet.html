Testing Strategy Cheat Sheet

The Testing Trophy

LayerCatchesCost
Static analysisType errors, typos, shape mismatchesLowest
UnitOne function/component's own logic, in isolationLow
IntegrationBugs that only appear when units combine (duplicated state, waterfalls)Medium — biggest confidence-per-cost
E2EReal-browser, close-to-production issuesHighest

Rule of thumb

"The more your tests resemble the way your software is used, the more confidence they can give you." — Kent C. Dodds. Mocking away the exact thing you're worried about (timing, shared state) makes a test pass without proving anything.

Architecture → testability, in one line each

  • Cohesive, deep components (Lesson 1/6) → cheap, meaningful unit tests.
  • Shared/lifted state (Lesson 2/7) → needs an integration test of the common ancestor.
  • Async Server Components (Lesson 3) → can't render() in isolation; favor integration/E2E.
  • Data waterfalls (Lesson 5) → invisible to instantly-resolved mocks; need real/realistic async timing.
